основная программа:
- balance_final.c - программа для исследования баланса вычислений и коммуникаций.
  эмулирует вычислительную нагрузку с помощью usleep и коммуникационную нагрузку 
  с помощью обмена сообщениями. позволяет настраивать соотношение вычислений к коммуникациям.

вспомогательный скрипт:
- run_final_experiments.sh - автоматизирует запуск серии экспериментов с разными 
  соотношениями вычислений и коммуникаций, а также разным количеством процессов.

алгоритм работы:

1. компиляция программы:
   mpicc balance_final.c -o balance_final

2. запуск экспериментов:
   ./run_final_experiments.sh

3. как работает программа:
   - каждый процесс получает свою порцию вычислений (total_compute / size)
   - объем коммуникаций пропорционален времени вычислений процесса
   - вычисления эмулируются с помощью usleep (микросекунды)
   - коммуникации выполняются по кольцевой топологии с помощью MPI_Sendrecv
   - измеряется общее время выполнения и вычисляются метрики производительности

4. параметры программы:
   - total_compute: общее время вычислений для одного процесса (секунды)
   - bytes_per_second: количество байт коммуникаций на секунду вычислений
   - iterations: количество итераций эксперимента для усреднения

5. тестируемые сценарии:
   - высокое соотношение (2000): много вычислений, мало коммуникаций
   - среднее соотношение (2): сбалансированный случай
   - низкое соотношение (0.002): мало вычислений, много коммуникаций
   - сравнение коммуникаций на одном узле vs на разных узлах

получаемые данные:
- время выполнения последовательной и параллельной версий
- ускорение (speedup) - во сколько раз быстрее работает параллельная версия
- эффективность - процент от идеального ускорения
- соотношение вычислений к коммуникациям (N/M)
- размер коммуникаций на процесс

анализ результатов:
- позволяет исследовать закон Амдала и влияние баланса вычислений/коммуникаций
- показывает как соотношение N/M влияет на масштабируемость
- демонстрирует разницу в производительности при внутриузловых и межузловых коммуникациях
- помогает определить оптимальное количество процессов для конкретного соотношения вычислений и коммуникаций
- иллюстрирует когда распараллеливание выгодно, а когда нет

ключевые выводы:
- при высоком соотношении N/M программа хорошо масштабируется
- при низком соотношении N/M добавление процессов может ухудшить производительность
- эффективность падает с ростом количества процессов из-за накладных расходов коммуникаций
- внутриузловые коммуникации значительно быстрее межузловых
